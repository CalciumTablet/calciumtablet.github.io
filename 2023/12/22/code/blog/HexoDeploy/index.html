简述 & 准备工作

简述

在 21 年上大学时一口气买了 3 年的腾讯云，当时感觉花了 150 真的还挺贵的，谁有知道现在的轻量服务器都能二三十一个月呢😓

其实静态 Blog 远比动态要好得多，不仅对性能要求更低，对带宽也很友好，不过也有个很大的问题就是编译的问题，每次发布新的内容都要重新编译一次再推送到服务器非常麻烦，不过有钱谁又不想搞一套高性能高带宽的 vps 呢😅

在互联网冲浪一周大概了解了 Github action 的用法，以及以前从未听过的 Vercle 的实力，又参考了许多前人的经验，最后写出了这篇文章，属于是缝缝补补


首先 Github 上很多服务都是免费的，并且限制也很少，Vercle 也有免费套餐，虽然有一定限制，但对于个人使用者来说也够用了，这里给不了解 Github 和 Vercle 的朋友讲一下大概的思路：

  • Github 建立公开静态库 (存放生成的静态文件) 及私人源码库 (存放 Hexo 源文件) 并生成 Token 用于本地与仓库之间的连接
  • 在本地配置好环境后部署 Hexo 并使用 Token 连接私人源码库
  • 配置私人源码库 Github Action 以及 公开静态库 Github Page
  • Vercle 导入公开静态库并绑定域名 (如果有)
  • Git 推送本地 Hexo 至私人源码库并等待 Github Action 编译后查看 Blog 运行情况

每次修改文件后的运行逻辑：
本地推送编辑好的 Hexo 至私人源码库 → 私人源码库通过 Github Action 生成静态文件并推送到公开静态库 → 公开静态库通过 Github Page 部署 Blog → Vercle 导入公开静态库并发布 Blog → 所有人通过 Vercle 发布的地址访问 Blog

整个过程只需要负责本地向仓库的推送，其他步骤全由 Github 与 Vercle 负责，每次推送文章只需要编辑完随手推送一下即可，简化操作专注写作


为什么要建两个仓库?

Github 公开仓库会毫无保留地向访者展示每一个文件每一行代码，这就会产生一个问题，你无法阻止有心者复制你的博客文件并稍作修改部署到自己的服务器上，如果对方在偷取你的文章的同时再积极偷一点其他人的文章那你们就成替身了🤣不仅如此，在后面步骤中还会编写 Workflows 文件，可能现在你还听不懂但只需要知道我们要将 Github Token 写入到这个文件，而 Token 的权限极大，那么有心者又多了一种选择，它可以破环你的 Blog
而私人仓库只有本人、被授权者或是使用 Key、Token 才能编辑修改，把源码存放在私人库里很安全，除非你的 Token 泄露或是账号被盗取，前者可能性不大，因为你自己都记不住 Token 😆，所以只需要注意保护好自己的账号就好了

准备工作

⭐必要的准备：

  • 一台操作系统是 Windows 的电脑 (如果你熟悉 Linux、MacOS 或是其他操作系统均可，这里只展示 Win 平台操作)，并安装好 Nodejs 和 Git
  • Github 账号 (免费注册)
  • 一个畅联全球的优质网络环境
  • 脑子🧠

⭐可选的准备：

  • 一个域名，更方便快捷的访问你的 Blog，配合 Vercle 一起使用
  • 一个代码编辑软件，更高效直观的编辑文件，推荐 VSCode (免费的放心用)
  • 一个 Md 编辑软件，便于编写文章，推荐 Typora (这个需要付费，也可以用 VSCode 装 Markdown 插件代替)

⭐可能用得上：

  • Github 官方文档
  • Vercle 官方文档
  • Hexo 官方文档
  • Nodejs 安装配置 Nodejs 使用介绍
  • Git 官方文档

最后调整好心态，遇到问题是很正常的。参考官方文档或是使用搜索引擎来解决问题，善用浏览器翻译和页面查找功能，积极解决问题






生成 Github Token

登入 Github 账号，前往 [Settings] → [Developer Settings]




前往 [Personal access tokens] → [Tokens (classic)] → [Generate new token] → [Generate new token (classic)]



来到 Token 生成页面， [Note] (Token 名称) 随意填就好， [Expiration] (有效期) 选择 [No expiration] (无期限)，下面在 [repo] 和 [workflow] 前面打钩，最后在最下面点击 [Generate token] 生成 Token

注意！不要将 Token 告诉他人，利用 Token 可以修改甚至删除你的文件！






建立 Github 仓库

通过右上角 [+] → [New Repository] 来创建仓库




创建私人源码仓库， [Repository name] (仓库名称) 可以随意填写， [Description] (仓库介绍) 选填， [Private] 打钩，最后点击 [Create repository] 创建私人仓库




创建公开静态仓库， [Repository name] (仓库名称) 填写 [{你的github用户名}.github.io] (我的用户名为 Calciumtablet，所以这里我填写 calciumtablet.github.io )， [Description] (仓库介绍) 选填， [Public] 和 [add a REDEME file] 打钩，最后点击 [Create repository] 创建公开仓库


创建好公开静态仓库后，前往库 [Settings] → [Pages] → Branch 选择 [Main] → [Save]






本地部署 Hexo

在此之前确保你的设备已经安装了 Nodejs 和 Git

在你喜欢的位置新建一个文件夹，名称随意，后面统一称这个文件夹为安装文件夹。注意，本地部署的 hexo 将会安装在这个位置，位置一旦选定不要轻易移动

Hexo 本地部署可以参考 Hexo 官方文档 开始使用部分，里面有多系统详细的安装教程

在新建的文件夹内打开命令行

  1. 输入 npm install -g hexo-cli 安装 Hexo 命令行工具
  2. 输入 hexo init catablog 创建 Blog 文件夹，后面统一称这一步创建的文件夹为 Blog 文件夹， catablog 是 Blog 文件夹名称，随意填写即可
  3. 输入 cd catablog 进入 Blog 文件夹， catablog 是 Blog 文件夹名称，与上一步相同
  4. 输入 npm install 安装所需依赖




安装好依赖后，本地部署就完成了，输入 hexo s 查看能否正常运行，正常运行效果如下图:


运行时我们可以通过 http://localhost:4000 来访问本地部署的 Hexo，想要停止运行在命令行同时按下 Ctrl+C 即可




如果觉得 Blog 有点简陋，可以通过安装 Hexo 主题 来美化,下面以 Redefine 主题为例操作一遍主题的安装配置

官方提供两种安装方法:

  • Blog 文件夹下输入命令: npm install hexo-theme-redefine@latest
  • Blog 文件夹下输入命令: git clone https://github.com/EvanNotFound/hexo-theme-redefine.git themes/redefine

两者最终效果相同，后者相较前者在修改上更加方便一些，如果你没有主题二次修改想法强烈建议以第一种方式安装，在需要更新时再次执行主题安装命令即可

注意！如果你通过 Git 安装主题并修改过主题文件，执行 Git 更新后会覆盖之前对文件的所有修改，切记更新前备份修改内容！


安装好主题后将 Blog 文件夹下的 _config.yml 文件中的 Theme: landscape 改为 Theme: 安装的主题名称 即可启用主题


再次运行 Hexo，访问 http://localhost:4000 看看效果如何，如果变化按 Crl+F5 强制刷新网页






连接 Github 仓库

Github 设置好了，本地也部署好了，接下来就是连接它们两个

  1. 输入 git init 初始化本地仓库
  2. 输入 git remote add origin https://{Token}@github.com/{Github用户名}/{私人库}.git 通过 Token 连接 Github 私人仓库
  3. 输入 git checkout -b main 更改分支为 Main 分支




本地添加用于 Github Action 自动化部署的 Workflows 文件，在 Blog文件夹/.github/workflows 目录下创建一个 autodeploy.yml 文件


将下面的代码标注部分替换后复制进去并保存，需要替换的内容有 [{Token} {Github用户名}/{Github用户名}] 在代码第 56/57 行




一切就绪，进行第一次推送

  1. 输入 git add . 添加所有文件到待提交列表
  2. 输入 git commit -m "Blog first push" 添加文件提交说明
  3. 输入 git push origin main 提交所有文件到 main 分支




推送成功后前往 Github 私人源码库 Action 选项卡确认自动部署是否正常运行，绿色表示无报错成功部署
确认成功部署后访问 https://{Github用户名}.github.io 确认一下博客是否正常加载，如果与在本地部署的 Blog 相同说明 Github page 也正常启动了
